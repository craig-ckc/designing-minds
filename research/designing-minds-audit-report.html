Full Website Audit

Designing Minds site audit and platform migration report

Audit of the public Designing Minds website at designingminds.co.za, including Home, About, Shop, Contact, Cart, Downloads, Privacy Policy, Terms route, sitemap, robots.txt, and a representative product page. The site is a WordPress, WooCommerce, Elementor build for CAPS-aligned printable tests. The recommended direction is a static-first Vite + React + Tailwind implementation deployed on Vercel.

Audit date: 19 June 2026 Scope: public site + shop flow sample Current stack: WordPress / PHP / Apache Target stack: Vite / React / Tailwind / Vercel

Executive Summary

C-Overall audit grade. The business offer is clear, but the site has heavy payloads, weak metadata, broken links, and accessibility structure issues.
86-93Stylesheet links on audited public pages. This is excessive for a content and digital-products site.
103-115Script tags on Home, About, Shop, Privacy, and representative product pages.
404The footer Terms & Conditions link resolves to a not-found page.
Critical

Static migration is justified for the whole public site

The public pages and product catalog are mostly structured content: grade categories, product cards, bundles, FAQs, policies, contact details, and checkout/download entry points. A Vite/React static build would remove PHP runtime dependency for most public pages, reduce asset weight, and simplify deployment. The only workflows needing backend design are payment, email, account/download access, and form submissions.

High

Reliability and link integrity need immediate fixes

The audit found browser console errors, a missing CSS asset, a broken Terms & Conditions route, and inconsistent server behavior during checks. The public site also exposes default WordPress sitemap entries, including user and builder-template sitemap indexes that are not ideal for a polished production ecommerce site.

Findings Breakdown

Critical

Performance: the whole site is carrying unnecessary page-builder and plugin overhead

Home, About, Shop, Privacy, and product pages all load broad WordPress, Elementor, WooCommerce, ShopEngine, product-filter, animation, carousel, lightbox, and icon-library assets. The privacy page alone is about 361 KB of HTML before assets, which is disproportionate for legal copy.

Recommendation: rebuild public pages as static routes, use route-level code splitting, generate CSS through Tailwind, and keep payment/cart/download scripts only on routes that need them.

High

SEO: core metadata is missing

The audited pages did not include meta descriptions, and there was no useful social/structured metadata detected in the fetched HTML. Titles exist, but product/category pages need richer search snippets and share previews.

Recommendation: define per-route metadata in a content config. Add Organization, WebSite, FAQPage, Product or Course-style structured data where appropriate. Use consistent canonical URLs.

High

Accessibility: heading hierarchy and icons need cleanup

Home, Contact, Privacy, and the sampled product page had no detected h1. Several logo/image links appear without accessible names in the browser snapshot, and some icon text renders as private-use glyphs. FAQ buttons are semantically buttons, which is good, but templates need consistent hierarchy and labels.

Recommendation: add one descriptive h1 per page, ensure logo links have aria-label, use SVG/icon components with accessible names hidden where decorative, and test keyboard focus states.

High

Commerce and downloads: the migration must preserve fulfillment

The business depends on selling digital printable tests and giving customers access to downloads. The current WordPress/WooCommerce setup handles cart, account, and downloads, but it does so with heavy public-page overhead.

Recommendation: before rebuilding checkout, document product SKUs, pricing, bundle rules, payment provider, fulfillment emails, refund policy, account requirements, and how downloadable files are protected.

High

Trust and conversion: contact and purchase flows need tightening

The contact details are visible and useful, but the form does not clearly explain expected response time, privacy handling, or submission success/error states in the static HTML. The newsletter field uses placeholder-style email text and no visible consent context. The broken Terms link harms trust.

Recommendation: add concise form helper text, validation, success/error states, privacy link, expected response time, and fix legal links before launch.

Medium

Sitemap quality: WordPress internals are being surfaced

The sitemap index includes user archives and builder/template content types such as Elementor/ElementsKit content, WPR templates, and ShopEngine templates. These are not usually pages customers should discover from search.

Recommendation: in the migration, generate a curated sitemap containing only real customer-facing routes, product routes, category routes, and policy pages.

Medium

Security and privacy headers are thin

The observed headers did not show modern hardening headers such as Content Security Policy, Strict-Transport-Security, X-Content-Type-Options, Referrer-Policy, or Permissions-Policy.

Recommendation: configure security headers in vercel.json. Start with a conservative CSP after confirming required third-party domains for analytics, payments, email, and social links.

Medium

Content: the value proposition is present but not sharpened

The footer states that the business supports home learning with high-quality CAPS-aligned practice materials. The FAQ makes the offer clearer than the page hero does. The contact page should reinforce who the site is for: South African parents, guardians, and learners needing printable CAPS-aligned Grade 3 to 7 tests.

Recommendation: add short, specific copy near the contact form: “Questions about printable CAPS-aligned tests for Grades 3 to 7? Send us a message.”

Working Well

Clear contact routes and useful FAQ topics

The phone, email, location, FAQ topics, My Downloads link, and Shop Tests navigation align with the business model. These should be preserved in the migration.

Migration Recommendation

The target architecture should be a static-first Vite React app with Tailwind CSS, generated to static files and deployed on Vercel. Keep public pages fast and simple; integrate payments, file delivery, and account/download workflows intentionally rather than reproducing WordPress plugin behavior by default.

Implementation Roadmap

  1. Discovery and content freeze: export page copy, FAQ answers, product list, pricing, test categories, images, policies, email templates, and download delivery rules from WordPress.
  2. Prototype static shell: create Vite, React, TypeScript, Tailwind, ESLint, and route structure. Build header/footer/contact page first because it is low-risk and representative.
  3. Design system pass: define typography, color tokens, spacing, buttons, form fields, cards, alerts, accordions, and focus states.
  4. SEO foundation: add route metadata, sitemap generation, robots.txt, canonical URLs, Open Graph images, FAQ structured data, and analytics.
  5. Commerce decision: either integrate a lightweight payment/download workflow or keep WooCommerce for shop/account temporarily behind subdomain or path separation.
  6. Vercel deployment: create preview deployments, add security headers, configure environment variables, connect domain, then run DNS cutover after verification.
  7. Post-launch checks: test forms, checkout, downloads, 404 page, legal pages, mobile nav, page speed, search indexing, and analytics events.

Vercel Deployment Checklist

  • Use npm create vite@latest designing-minds -- --template react-ts or equivalent.
  • Add Tailwind and centralize brand tokens in tailwind.config.
  • Generate static HTML for all public routes before deployment.
  • Add vercel.json for redirects, clean URLs, and security headers.
  • Set environment variables for form email, payment provider keys, analytics, and download signing secret.
  • Add redirects from old WordPress URLs: /about-3/ to /about/, /cart-2/ to /cart/, and fixed legal routes.
  • Run Lighthouse/PageSpeed, keyboard navigation checks, form submission tests, and checkout/download tests before DNS cutover.
